## What's Done — Phase by Phase

### ✅ Phase 1 — Project bootstrap
- Flutter Android project, bundle id `com.salawat.app`
- Light + dark themes, Cairo font (google_fonts), full RTL
- Riverpod state, `shared_preferences` for local persistence
- Navigation shell: 3 bottom tabs (الرئيسية / لوحة الشرف / إنجازاتي), gradient header
- Files: `mobile/lib/main.dart`, `app.dart`, `theme/`, `shared/nav_shell.dart`

### ✅ Phase 2 — Onboarding + offline counter
- Onboarding card (gradient bg, name input, "توكلنا على الله") → routes to nav shell
- Home: 256×256 tap button with ripple, decorative rotating rings, Arabic-Indic counter, "صَلِّ عَلَيْهِ" label
- 8-badge gamification ladder (مبتدئ → الشفاعة المرجوة, 10 → 100,000)
- Badge unlock SnackBar + heavy haptic
- Files: `mobile/lib/features/onboarding/`, `features/home/`, `models/badge.dart`, `core/counter_controller.dart`

### ✅ Phase 3 — Firebase + Cloud Function + sync
- `Firebase.initializeApp` + Anonymous Auth on startup (`AuthGate`)
- `CounterSync` service: batches deltas every 2.5s, persists `pendingReqId` for cross-restart idempotency, flushes on app pause
- Cloud Function `incrementCount` (TypeScript, v2 callable, Node 22, `maxInstances: 10`):
  - Validates auth, delta range 1–1000, clientReqId
  - Idempotency table (`idempotency/{reqId}` with 24h TTL)
  - Single batch write: random global shard + lifetime LB + daily LB + user profile + idem record
- Firestore rules lock all counter docs from client writes; only the Admin SDK function can modify
- Files: `mobile/lib/data/`, `functions/src/incrementCount.ts`, `firestore.rules`

### ✅ Phase 4 — Leaderboard tab
- Top 50 streamed via `orderBy('count', desc).limit(50).snapshots()`
- "My rank" via `count() WHERE count > myCount` aggregation, **throttled** to once per 5s and only on count delta > 5 (prevents read storms)
- Sticky bottom card shown when user's rank > 50
- Pull-to-refresh
- Files: `mobile/lib/features/leaderboard/`, `data/leaderboard_repository.dart`

### ✅ Phase 5 — Profile tab

- 8-badge grid with locked/unlocked states + per-badge progress bar when locked
- Files: `mobile/lib/features/profile/`

### ✅ Phase 6 — Polish

## Recommendations

### 🔴 Do Soon (within a week)

1. **Verify production end-to-end on a real device.** Code is deployed but never tested against the live backend. Likely-but-unconfirmed failure modes: cold-start latency, Firestore index build delays, region mismatch between Firestore and Function.

2. **Set Artifact Registry cleanup policy** (saves ~$0.50/mo as deploys accumulate):
   ```powershell
   firebase functions:artifacts:setpolicy --location=us-central1 --days=1
   ```

3. **Sort out repo ownership.** The repo is on `ahmed635`'s personal GitHub account. The customer's app shouldn't live on a single dev's account long-term:
   - Minimum: add the customer as a collaborator (Settings → Collaborators)
   - Better: transfer the repo to a customer-owned org, keep your fork for development

4. **Confirm Firestore region.** Was supposed to be `eur3` (good for MENA) but verify in the console (Firestore → Settings). If it's `nam5`, latency to MENA users will be ~150ms higher, and **the region cannot be changed** — would need a new project to fix.

### 🟡 Do Before Launch

5. **App icon + splash screen.** Launcher currently shows the default Flutter "F". Hard ship-blocker for the Play Store.

6. **Add App Check.** Free, ~30 minutes. Blocks 95% of casual abuse — without it, anyone with the project ID can call `incrementCount` from `curl` once they know the URL.

7. **Privacy policy.** Required by Play Store for any app that collects user data (Firebase Auth = data collection). The customer needs to provide one or approve a generated template.

8. **Spiritual / ethical review** — *important and easy to overlook*. Ranked leaderboards in a religious app can encourage **riya'** (worship for show). Pre-launch, talk to a scholar in the target community about whether to:
   - Switch global leaderboard to **tier display** ("Top 5%") instead of exact rank
   - Default to **anonymous** on the leaderboard (opt-in to show name)
   - Make **group leaderboards** (family, masjid, friends) the primary social mode
   - Emphasise the **community total** over individual ranking
   - This is the §10 discussion from the original `docs/ARCHITECTURE.md`. The current React-derived UI doesn't reflect it yet.

### 🟢 Longer-term

9. **Cross-device sync.** A user reinstalling today loses their lifetime count locally even though it's still in Firestore. Add: on sign-in, fetch `leaderboardLifetime/{uid}.count` and adopt if higher than the local prefs value. ~2 hours.

10. **Onboarding name validation.** No length cap, no profanity filter — offensive names appear on the leaderboard. Add server-side filter (callable Cloud Function or Firestore rules check) before launch.

11. **"Today" vs "lifetime" — pick one.** Home button labels the count "صلاة اليوم" (today) but actually shows lifetime. Either rename the label, or actually track today's count locally + reset daily. Decide alongside the daily reset feature in Phase 7.

12. **iOS support.** Currently Android-only. Adds: Apple Dev account ($99/yr), separate FlutterFire configure for iOS, APNS for push, separate Apple App Store review process. ~1 week additional work.

13. **Monitoring beyond the budget alert.**
    - Firebase **Crashlytics** for crash reports (free)
    - Cloud Logging dashboard for function errors
    - Uptime alert (e.g. Better Uptime free tier)

14. **CI/CD.** GitHub Actions workflow for PR checks (`flutter analyze` + `flutter test`) and tag-triggered Play Store deploy.

15. **Scale planning beyond 5K DAU.** The "rank by count() aggregation" approach in `LeaderboardRepository` costs ~1 read per 1000 docs above the user — gets pricey at scale. Switch to tier display ("Top 1%") computed from a periodic snapshot. Aligns with recommendation #8 too.

## Cost Summary

- **Expected at 100–1000 DAU: ~$0/month** (Blaze free tier covers it: 50K reads, 20K writes, 2M function invocations daily).
- **First chargeable threshold: ~5K DAU** — will start to dribble cents per day for write overflow.
- **Spike risks**: a leaderboard listener storm (mitigation: paginate), runaway function instances (mitigation: `maxInstances: 10`), frequent rank queries (mitigation: throttling, already in place).
- **The $5/month budget alert** will email the customer if anything goes wrong. **Verify it's actually configured** at <https://console.cloud.google.com/billing/budgets?project=salawat-e1098>.
